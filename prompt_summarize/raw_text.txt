ข้อเสนอแนะจากการตรวจสอบโค้ด

กลยุทธ์การทดสอบ QA

เริ่มกันที่การอัพเดตความคืบหน้าของสปรินท์ ซาร่าห์ ในฐานะนักพัฒนาอาวุโสหลักของเรา คุณช่วยให้ภาพรวมได้ไหม?
นักพัฒนาอาวุโส (ซาร่าห์): ขอบคุณค่ะ PM เราผ่านมาสองสัปดาห์แล้วในสปรินท์หกสัปดาห์ของเรา และเราได้ทำความคืบหน้าที่ดีในฟังก์ชันหลัก แคตตาล็อกสินค้าและฟีเจอร์การค้นหาเสร็จไปแล้วประมาณ 70% เราได้ใช้งานสคีมาฐานข้อมูลพื้นฐานและตั้งค่า API endpoints เบื้องต้นแล้ว อย่างไรก็ตาม เรายังล่าช้าเล็กน้อยในระบบยืนยันตัวตนของผู้ใช้
นักพัฒนาจูเนียร์ (อเล็กซ์): ขอโทษนะครับ ซาร่าห์ คุณช่วยอธิบายได้ไหมครับว่าอะไรทำให้เกิดความล่าช้าในระบบยืนยันตัวตน? ผมกำลังทำงานกับการผสานรวมฝั่งหน้าบ้าน และอยากให้แน่ใจว่าผมไม่ได้ทำให้เกิดการติดขัด
ซาร่าห์: คำถามดีนะคะ อเล็กซ์ เรากำลังถกเถียงกันว่าจะใช้ระบบยืนยันตัวตนของเราเองหรือใช้บริการของบุคคลที่สามอย่าง Auth0 การตัดสินใจนี้ใช้เวลานานกว่าที่คาดไว้ แต่มันสำคัญมากสำหรับความปลอดภัยและความสามารถในการขยายตัว
PM: ขอบคุณสำหรับการอัพเดต เรามาต่อกันที่ความท้าทายทางเทคนิค จอห์น คุณได้กล่าวถึงความกังวลเกี่ยวกับสถาปัตยกรรมแบ็กเอนด์ของเราใช่ไหม?
นักพัฒนาอาวุโส (จอห์น): ใช่ครับ เราต้องตัดสินใจระหว่างสถาปัตยกรรมไมโครเซอร์วิสและแบบโมโนลิธิก เมื่อพิจารณาจากกำหนดเวลาและขนาดทีมของเรา ผมโน้มเอียงไปทางโมโนลิธแบบโมดูลาร์ มันจะง่ายกว่าในการพัฒนาและติดตั้งในตอนแรก และเราสามารถแยกออกเป็นไมโครเซอร์วิสได้ในภายหลังถ้าจำเป็น
ซาร่าห์: ฉันไม่เห็นด้วยค่ะ จอห์น แม้ว่าโมโนลิธอาจจะเร็วกว่าในตอนเริ่มต้น แต่ฉันคิดว่าเราจะได้ประโยชน์จากไมโครเซอร์วิสในระยะยาว โดยเฉพาะสำหรับแพลตฟอร์ม e-commerce ที่ต้องการปรับขนาดส่วนประกอบต่างๆ อย่างอิสระ
PM: นี่เป็นการตัดสินใจที่สำคัญ เรามาฟังความคิดเห็นจากนักพัฒนาอาวุโสอีกคนของเรา ไมค์
นักพัฒนาอาวุโส (ไมค์): ผมเห็นประเด็นทั้งสองฝ่าย ถ้าเราใช้วิธีแบบผสมผสานล่ะครับ? เราสามารถเริ่มต้นด้วยโมโนลิธแบบโมดูลาร์ แต่ออกแบบให้มีขอบเขตที่ชัดเจนซึ่งจะทำให้ง่ายขึ้นในการแยกเป็นไมโครเซอร์วิสในภายหลัง
ผู้เชี่ยวชาญ QA (เอ็มม่า): จากมุมมองการทดสอบ ฉันชอบแนวทางของไมค์ มันจะช่วยให้เราสามารถตั้งค่าการทดสอบการผสานรวมที่ครอบคลุมได้ตั้งแต่เริ่มต้น ในขณะที่ยังคงเปิดโอกาสสำหรับการทดสอบไมโครเซอร์วิสที่ละเอียดขึ้นในอนาคต
PM: ฟังดูเป็นการประนีประนอมที่ดีนะ เรามาใช้ข้อเสนอของไมค์และทบทวนมันอีกครั้งตอนจบสปรินท์นี้ ตอนนี้ เรามาพูดถึงข้อเสนอแนะจากการตรวจสอบโค้ด อเล็กซ์ ผมเชื่อว่าคุณมีคำถามบางอย่าง?
อเล็กซ์: ครับ ผมสังเกตเห็นว่าโค้ดฝั่งหน้าบ้านของเราเป็นการผสมผสานระหว่างคอมโพเนนต์ของ React และ Vue มันกำลังกลายเป็นเรื่องยากที่จะบำรุงรักษา เราควรจะมาตรฐานเป็นเฟรมเวิร์คเดียวไหมครับ?
ซาร่าห์: คุณพูดถูกค่ะ อเล็กซ์ ความไม่สอดคล้องกันนี้เกิดขึ้นเพราะเราเริ่มต้นด้วย React แต่สมาชิกบางคนในทีมถนัด Vue มากกว่า เราควรจะมาตรฐานเป็น React เพราะมันมีประสิทธิภาพที่ดีกว่าสำหรับกรณีการใช้งานของเรา และมีระบบนิเวศที่ใหญ่กว่า
จอห์น: ผมเห็นด้วยกับการทำให้เป็นมาตรฐานเดียว แต่ผมคิดว่าเราควรพิจารณา Vue ด้วย การเรียนรู้มันง่ายกว่า ซึ่งอาจจะเป็นประโยชน์สำหรับนักพัฒนาจูเนียร์ของเรา
PM: เรามาพักเรื่องนี้ไว้ก่อน และกำหนดการประชุมตัดสินใจด้านสถาปัตยกรรมแยกต่างหาก ซาร่าห์ กรุณาเตรียมการเปรียบเทียบระหว่าง React กับ Vue สำหรับความต้องการของโครงการเรา
ตอนนี้ เรามาจัดลำดับความสำคัญของฟีเจอร์ที่กำลังจะมาถึง ลูกค้าของเราต้องการให้เราโฟกัสที่ตะกร้าสินค้าและกระบวนการชำระเงินเป็นลำดับถัดไป มีข้อพิจารณาทางเทคนิคอะไรที่เราควรระวังไหม?
ไมค์: เราต้องแน่ใจว่าฐานข้อมูลของเราสามารถจัดการกับธุรกรรมที่เกิดขึ้นพร้อมกันได้อย่างมีประสิทธิภาพ ผมขอเสนอให้เราใช้การล็อคแบบ optimistic เพื่อจัดการกับความขัดแย้งที่อาจเกิดขึ้นระหว่างการชำระเงิน
เอ็มม่า: จากมุมมองของ QA เราจะต้องทดสอบการผสานรวมเกตเวย์การชำระเงินต่างๆ อย่างละเอียด เราสามารถจัดลำดับความสำคัญในการตั้งค่าสภาพแวดล้อม sandbox สำหรับเรื่องนี้ได้ไหมคะ?
PM: ประเด็นที่ดีมาก เอ็มม่า เรามาทำให้มันเป็นความสำคัญลำดับต้นๆ สำหรับสปรินท์หน้า สุดท้าย เรามาพูดถึงกลยุทธ์การทดสอบ QA เอ็มม่า แผนของคุณเป็นอย่างไรบ้าง?
เอ็มม่า: ฉันวางแผนที่จะใช้การทดสอบแบบผสมผสานทั้งการทดสอบหน่วย การทดสอบการผสานรวม และการทดสอบแบบ end-to-end โดยใช้ Jest และ Cypress เราควรพิจารณาการใช้การทดสอบสัญญา (contract testing) สำหรับ API ของเราด้วย เพื่อให้แน่ใจว่าการผสานรวมระหว่างหน้าบ้านและหลังบ้านราบรื่น
ซาร่าห์: ฟังดูครอบคลุมดีค่ะ เราสามารถรวมการทดสอบประสิทธิภาพด้วยได้ไหม โดยเฉพาะสำหรับฟังก์ชันการค้นหาสินค้า?
เอ็มม่า: ได้เลยค่ะ ฉันจะเพิ่มมันลงในแผนการทดสอบของเรา
