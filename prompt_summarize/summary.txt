การอัพเดตความคืบหน้าของสปรินท์เผยว่า ทีมพัฒนาสปรินท์หกสัปดาห์ได้ทำความคืบหน้าที่ดีในฟังก์ชันหลัก แคตตาล็อกสินค้าและฟีเจอร์การค้นหาเสร็จไปแล้วประมาณ 70% และได้ใช้งานสคีมาฐานข้อมูลพื้นฐานและตั้งค่า API endpoints เบื้องต้นแล้ว อย่างไรก็ตาม ทีมยังล่าช้าเล็กน้อยในระบบยืนยันตัวตนของผู้ใช้ ซึ่งอาจเกิดจากความท้าทายทางเทคนิคในการพัฒนาระบบยืนยันตัวตน.
ทีมกำลังพิจารณาว่าจะใช้ระบบยืนยันตัวตนของตนเองหรือใช้บริการของบุคคลที่สามอย่าง Auth0 ซึ่งการตัดสินใจนี้ใช้เวลานานกว่าที่คาดไว้ แต่มันสำคัญมากสำหรับความปลอดภัยและความสามารถในการขยายตัว นอกจากนี้ยังมีความท้าทายทางเทคนิคในการเลือกสถาปัตยกรรมแบ็กเอนด์ระหว่างไมโครเซอร์วิสและแบบโมโนลิธิก โดยนักพัฒนาอาวุโส (จอห์น) โน้มเอียงไปทางโมโนลิธแบบโมดูลาร์ เนื่องจากมันจะง่ายกว่าในการพัฒนาและติดตั้งในตอนแรก
ทีมกำลังพิจารณาว่าจะใช้โมโนลิธแบบโมดูลาร์หรือไมโครเซอร์วิสในการพัฒนาแพลตฟอร์ม e-commerce จอห์นชื่นชอบโมโนลิธเพราะมันง่ายกว่าในการพัฒนาและติดตั้งในตอนแรก แต่ซาร่าห์เห็นว่าไมโครเซอร์วิสจะให้ประโยชน์ในระยะยาวโดยเฉพาะสำหรับการปรับขนาดส่วนประกอบต่างๆ อย่างอิสระ ไมค์ นักพัฒนาอาวุโสเสนอแนวทางผสมผสานที่เริ่มต้นด้วยโมโนลิธแบบโมดูลาร์แต่ออกแบบให้มีขอบเขตที่ชัดเจนเพื่อง่ายต่อการแยกเป็นไมโครเซอร์วิสในภายหลัง เอ็มม่า ผู้เชี่ยวชาญ QA ชื่นชอบแนวทางของไมค์
ทีมกำลังพูดคุยเกี่ยวกับการทดสอบและการตรวจสอบโค้ด ในมุมมองการทดสอบ ไมค์เสนอแนวทางที่จะช่วยให้สามารถตั้งค่าการทดสอบการผสานรวมที่ครอบคลุมได้ตั้งแต่เริ่มต้น และเปิดโอกาสสำหรับการทดสอบไมโครเซอร์วิสที่ละเอียดขึ้นในอนาคต ทีมตกลงที่จะใช้ข้อเสนอของไมค์และทบทวนมันอีกครั้งตอนจบสปรินท์นี้ นอกจากนี้ อเล็กซ์ยังมีข้อเสนอแนะจากการตรวจสอบโค้ด โดยสังเกตเห็นว่าโค้ดฝั่งหน้าบ้านของพวกเขามีการผสมผสานระหว่างคอมโพเนนต์ของ React และ Vue ซาร่าห์เห็นด้วยว่าควรจะมาตรฐานเป็นเฟรมเวิร์คเดียว คือ React.
ทีมกำลังพิจารณาว่าจะใช้ React หรือ Vue เป็นมาตรฐานสำหรับโครงการของพวกเขา จอห์นเสนอให้พิจารณา Vue ด้วย เนื่องจากมันง่ายต่อการเรียนรู้ ซึ่งอาจจะเป็นประโยชน์สำหรับนักพัฒนาจูเนียร์ แต่การตัดสินใจนี้ถูกเลื่อนออกไปเพื่อประชุมตัดสินใจด้านสถาปัตยกรรมแยกต่างหาก

ทีมจึงหันไปจัดลำดับความสำคัญของฟีเจอร์ที่กำลังจะมาถึง โดยลูกค้าต้องการให้โฟกัสที่ตะกร้าสินค้าและกระบวนการชำระเงินเป็นลำดับถัดไป ไมค์เตือนให้แน่ใจว่าฐานข้อมูลสามารถจัดการกับธุรกรรมที่เกิดขึ้นพร้อมกันได้อย่างมีประสิทธิภาพ และเสนอให้ใช้การล็อคแบบ optimistic
ทีมกำลังหารือเกี่ยวกับการจัดการความขัดแย้งที่อาจเกิดขึ้นระหว่างการชำระเงิน โดยเสนอการใช้การล็อคแบบ optimistic และการผสานรวมเกตเวย์การชำระเงินต่างๆ ทีม QA จะทดสอบการผสานรวมนี้อย่างละเอียด โดยใช้การทดสอบแบบผสมผสานทั้งการทดสอบหน่วย, การทดสอบการผสานรวม, และการทดสอบแบบ end-to-end ด้วย Jest และ Cypress และพิจารณาการใช้การทดสอบสัญญา (contract testing) สำหรับ API เพื่อให้แน่ใจว่าการผสานรวมระหว่างหน้าบ้านและหลังบ้านราบรื่น ทีมยังพิจารณารวมการทดสอบประสิทธิภาพด้วย โดยเฉพาะสำหรับฟังก์ชันการค้นหาสินค้า.
